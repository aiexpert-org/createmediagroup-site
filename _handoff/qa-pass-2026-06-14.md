## Issues found and fixed in this commit

### 1. (HIGH) Per-tile mosaic hover never triggered — gradient overlay swallowed all pointer events
- **Where:** `HeroMosaicBackground` (home hero + now /how-it-works hero).
- **Symptom:** The Studio per-panel color pop was implemented on every tile, but
  hovering the mosaic did nothing. `document.elementFromPoint()` at a visible
  right-side tile's center returned `div.absolute.inset-0` (the right-to-left
  white gradient overlay) — not the tile. The full-bleed gradient and the
  bottom-fade div sat above the tiles and intercepted every hover.
- **Fix:** added `pointer-events-none` to both overlay divs. After the fix,
  `elementFromPoint` returns the tile `img`, and hovering a right-side tile
  transitions `grayscale(0.85) brightness(0.95) opacity .55` →
  `grayscale(0) brightness(1.05) opacity 1` + `scale(1.04)` (verified via
  computed styles in the real browser).
- This is the exact "so poorly executed" class of bug the QA pass was meant to
  catch: it looked done in code but did nothing on screen.

### 2. (HIGH) Dark-background CTAs rendered dark text over the yellow marker (FIX 5)
- **Where:** `ContactBlock` (home, portfolio, subscription, how-it-works, every
  article) AND the subscription **featured** PriceCard (`bg-neutral-950`).
- **Symptom:** `Button` text was a fixed near-black across all backgrounds; on the
  black panels the letters disappeared under the opaque yellow marker.
- **Fix:** added a `tone="light" | "dark"` prop to `Button`. `dark` flips text to
  white (marker stays yellow). Wired `tone="dark"` into `ContactBlock` and the
  featured PriceCard. Verified white (`rgb(255,255,255)`) over the yellow marker
  on both, fully legible.

## Tooling note (why the real browser, not Claude Preview)

The Claude Preview pane reported false hover state: it persisted an emulated
cursor over the trust-band logos, so their `group-hover` (yellow) variant showed
even with nothing hovered, and computed styles for those `img`s read
white=0 / yellow=1. The nav markers (not under that cursor) read correctly, which
is what exposed the discrepancy. In real Chromium (no forced hover) the same
logos read white=1 / yellow=0 — correct. All hover/contrast verification in this
doc was therefore done in `browse`, not the preview pane.
